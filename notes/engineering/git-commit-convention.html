Git 提交规范指南 ​

规范概述 ​

为什么需要提交规范？ ​

统一的 Git 提交规范能够带来以下好处：

  • 提高可读性：清晰的提交信息便于快速理解代码变更
  • 自动化工具：规范的提交信息可以自动生成 CHANGELOG
  • 版本管理：基于提交类型自动确定版本号（语义化版本）
  • 代码审查：帮助审查者快速了解变更内容
  • 问题追踪：便于定位和回溯问题

规范标准 ​

本规范基于 Conventional Commits 标准，这是目前最广泛采用的提交规范。

提交格式规范 ​

基本格式 ​

<type>(<scope>): <subject>

<body>

<footer>

格式说明 ​

Header（必需） ​

格式：<type>(<scope>): <subject>

  • type（必需）：提交类型，见下方详细说明
  • scope（可选）：影响范围，如模块名、组件名等
  • subject（必需）：简短描述，不超过 50 个字符

Body（可选） ​

详细描述本次提交的内容，可以包括：

  • 变更的动机
  • 与之前版本的对比
  • 相关的 issue 或 PR

要求：

  • 使用第一人称现在时，如 "change" 而不是 "changed" 或 "changes"
  • 每行不超过 72 个字符
  • 空行分隔 Header 和 Body

Footer（可选） ​

用于记录：

  • Breaking Changes：不兼容的变更
  • 关闭 Issue：如 Closes #123, #456
  • 相关链接：如 Refs #789

示例 ​

简单提交 ​

feat: 添加用户登录功能

提交范围说明 ​

范围的作用 ​

范围（scope）用于说明提交影响的部分，可以是：

  • 模块名：auth、user、order
  • 组件名：Button、Header、LoginForm
  • 文件名：router.js、api.ts
  • 功能区域：ui、api、utils

范围命名规范 ​

  • 使用小写字母
  • 使用连字符分隔多个单词
  • 保持简洁，通常 1-2 个单词

示例 ​

bash
feat(auth): 添加双因素认证
fix(ui-button): 修复按钮点击事件
docs(api): 更新用户接口文档
refactor(utils): 重构日期格式化函数
1
2
3
4

何时使用范围 ​

  • 推荐使用：当提交影响特定模块或组件时
  • 可以不使用：当提交影响多个模块或整个项目时
bash
# 影响特定模块，使用范围
feat(auth): 添加登录功能

# 影响整个项目，不使用范围
chore: 更新项目依赖

    "subject-empty": [2, "never"],
    "subject-full-stop": [2, "never", "."],
    "type-case": [2, "always", "lower-case"],

常见问题 ​

Q1: 如何修改上一次提交？ ​

如果只是修改提交消息：

bash
git commit --amend
1

如果需要修改提交内容：

bash
# 修改文件后
git add .
git commit --amend --no-edit
1
2
3

Q2: 如何合并多个提交？ ​

使用交互式 rebase：

bash
git rebase -i HEAD~3  # 合并最近 3 个提交
1

在编辑器中，将需要合并的提交标记为 squash 或 s。

Q3: 提交类型选错了怎么办？ ​

如果还没有推送：

bash
git commit --amend
1

如果已经推送：

bash
git commit --amend
git push --force  # 注意：强制推送需要谨慎使用
1
2

Q4: 如何为提交添加多个范围？ ​

通常一个提交只影响一个范围，如果确实影响多个范围，可以：

  1. 不使用范围：feat: 添加新功能
  2. 使用通用范围：feat(core): 添加新功能
  3. 拆分成多个提交

Q5: 提交消息可以使用中文吗？ ​

可以，但建议：

  • 团队内部项目：可以使用中文，提高可读性
  • 开源项目：建议使用英文，便于国际化
  • 企业项目：根据团队规范决定

Q6: 如何处理紧急修复？ ​

紧急修复也应该遵循规范：

bash
fix(hotfix): 紧急修复支付接口超时问题

支付接口在高并发情况下出现超时，导致订单创建失败。
临时解决方案：增加超时时间和重试机制。

Q8: 如何处理重构提交？ ​

重构应该：

  1. 明确说明重构原因：为什么需要重构
  2. 说明重构内容：具体做了哪些改动
  3. 确保功能不变：重构不应该改变功能行为
bash
refactor(api): 重构用户服务层

原因：
- 代码结构混乱，难以维护
- 存在重复逻辑
- 缺乏错误处理

改动：
- 提取了公共方法
- 统一了错误处理
- 优化了代码结构
- 添加了类型定义

功能保持不变，仅优化代码结构。

最佳实践总结 ​

✅ 推荐做法 ​

  1. 提交前检查：使用工具自动检查提交格式
  2. 及时提交：完成一个小功能就提交，不要积累大量变更
  3. 清晰描述：提交消息要清晰描述变更内容
  4. 关联 Issue：提交时关联相关的 Issue
  5. 拆分大型提交：大型功能拆分成多个小提交
  6. 使用工具：使用 Commitizen 等工具辅助生成提交消息

❌ 避免做法 ​

  1. 模糊描述：避免使用 "修复 bug"、"更新代码" 等模糊描述
  2. 混合变更：不要在同一个提交中混合不同类型的变更
  3. 过长的提交：避免一次提交包含大量不相关的变更
  4. 忽略规范：不要因为紧急就忽略提交规范
  5. 强制推送：谨慎使用 git push --force，避免影响团队协作
