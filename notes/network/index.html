网络知识体系 ​

网络基础 ​

网络是什么 ​

  • 网络是连接多台计算机的系统，允许它们之间进行数据交换和资源共享
  • 互联网（Internet）是全球最大的计算机网络，由无数个局域网、城域网和广域网互联而成
  • 万维网（WWW）是建立在互联网之上的信息空间，使用 HTTP/HTTPS 协议进行通信

网络分层 ​

OSI 七层模型 ​

  1. 物理层：传输比特流，定义电气特性和物理连接
  2. 数据链路层：帧的传输，错误检测和纠正
  3. 网络层：路由选择，IP 协议
  4. 传输层：端到端通信，TCP/UDP 协议
  5. 会话层：建立、管理和终止会话
  6. 表示层：数据格式转换、加密解密
  7. 应用层：HTTP、FTP、SMTP 等应用协议

TCP/IP 四层模型 ​

  1. 网络接口层：对应 OSI 的物理层和数据链路层
  2. 网络层：IP 协议，负责路由和寻址
  3. 传输层：TCP/UDP 协议，负责端到端通信
  4. 应用层：HTTP、HTTPS、FTP、DNS 等

五层模型（常用） ​

  1. 物理层
  2. 数据链路层
  3. 网络层（IP）
  4. 传输层（TCP/UDP）
  5. 应用层（HTTP/HTTPS）

HTTP/HTTPS ​

HTTP 协议 ​

  • 超文本传输协议（HyperText Transfer Protocol）
  • 无状态协议，每次请求都是独立的
  • 基于请求-响应模式
  • 默认端口：80

HTTPS 协议 ​

  • 安全超文本传输协议（HyperText Transfer Protocol Secure）
  • 在 HTTP 基础上加入 SSL/TLS 加密层
  • 提供数据加密、身份认证和数据完整性保护
  • 默认端口：443
  • 使用数字证书验证服务器身份

HTTPS/TLS 握手过程（四次握手） ​

HTTPS 在建立连接前需要进行 TLS 握手，这个过程通常被称为"四次握手"：

第一次握手（Client Hello）：

  • 客户端向服务器发送：
    • 支持的 TLS 版本（如 TLS 1.2、TLS 1.3）
    • 支持的加密套件列表（Cipher Suites）
    • 支持的压缩方法
    • 随机数（Client Random），用于后续密钥生成
    • 会话 ID（如果之前有会话，用于会话恢复）
    • 服务器名称（SNI，Server Name Indication），用于虚拟主机

第二次握手（Server Hello）：

  • 服务器向客户端发送：
    • 选择的 TLS 版本
    • 选择的加密套件
    • 随机数（Server Random），用于后续密钥生成
    • 会话 ID
    • 数字证书（包含公钥、服务器信息、CA 签名）
    • 服务器密钥交换（如果需要，如 ECDHE）
    • 证书请求（如果需要客户端证书）
    • Server Hello Done

客户端验证证书：

  • 验证证书是否过期
  • 验证证书是否由可信 CA 签发
  • 验证证书中的域名是否匹配
  • 验证证书链是否完整
  • 如果验证通过，提取服务器的公钥

第三次握手（Client Key Exchange）：

  • 客户端生成预主密钥（Pre-Master Secret）
  • 使用服务器的公钥加密预主密钥，发送给服务器
  • 客户端发送：
    • 加密的预主密钥
    • 改变加密规范通知（Change Cipher Spec）
    • 客户端握手完成（Client Finished），使用协商的密钥加密

第四次握手（Server Finished）：

  • 服务器使用私钥解密预主密钥
  • 服务器和客户端各自使用预主密钥、Client Random、Server Random 生成主密钥（Master Secret）
  • 服务器发送：
    • 改变加密规范通知（Change Cipher Spec）
    • 服务器握手完成（Server Finished），使用协商的密钥加密

握手完成：

  • 双方使用主密钥生成会话密钥
  • 后续通信使用对称加密（AES 等），性能更好
  • 连接建立完成，可以开始传输 HTTP 数据

TLS 握手的关键点：

  1. 非对称加密：握手阶段使用非对称加密（RSA、ECDHE 等）传输密钥
  2. 对称加密：数据传输阶段使用对称加密（AES 等），性能更好
  3. 数字证书：验证服务器身份，防止中间人攻击
  4. 前向保密：使用 ECDHE 等算法可以实现前向保密，即使私钥泄露，历史通信也无法解密

TLS 1.3 的改进：

  • 减少握手次数（1-RTT 或 0-RTT）
  • 更快的连接建立
  • 更强的安全性
  • 移除不安全的加密套件

HTTP 版本演进 ​

  • HTTP/1.0：每个请求需要建立新连接
  • HTTP/1.1：支持持久连接、管道化、分块传输
  • HTTP/2：多路复用、头部压缩、服务器推送
  • HTTP/3：基于 QUIC 协议，解决队头阻塞问题

URL 组成部分详解 ​

1. 协议（Protocol/Scheme） ​

  • HTTP：http://，非加密传输
  • HTTPS：https://，加密传输
  • 其他：ftp://、ws://、wss://、file:// 等

2. 域名（Domain/Host） ​

  • 服务器的地址，如 www.example.com
  • 可以是 IP 地址，如 192.168.1.1

3. 端口（Port） ​

  • 可选，默认 HTTP 为 80，HTTPS 为 443
  • 格式：:8080

4. 路径（Path） ​

  • 资源在服务器上的路径
  • 如：/api/users/123、/products/item.html

5. 查询参数（Query Parameters） ​

  • URL 中 ? 后面的部分
  • 格式：?key1=value1&key2=value2
  • 多个参数用 & 连接
  • 值需要进行 URL 编码（encodeURIComponent）

示例：

https://www.example.com/search?q=javascript&page=1&limit=20
1

Query 参数的特点：

  • 在 URL 中可见
  • 会出现在浏览器地址栏
  • 可以通过 window.location.search 获取
  • 会出现在 HTTP 请求的 URL 中
  • 可以用于 GET 请求传递数据
  • 会被浏览器缓存和历史记录保存

JavaScript 操作 Query 参数：

javascript
// 获取查询参数
function getQueryParams() {
  const params = new URLSearchParams(window.location.search);
  return {
    q: params.get('q'),
    page: params.get('page'),
    limit: params.get('limit')
  };
}

// 或者手动解析
function parseQueryString(queryString) {
  const params = {};
  queryString.substring(1).split('&').forEach(param => {
    const [key, value] = param.split('=');
    params[decodeURIComponent(key)] = decodeURIComponent(value || '');
  });
  return params;
}

// 构建查询字符串
function buildQueryString(params) {
  const queryString = Object.keys(params)
    .map(key => `${encodeURIComponent(key)}=${encodeURIComponent(params[key])}`)
    .join('&');
  return queryString ? `?${queryString}` : '';
}

// 使用示例
const params = { q: 'javascript', page: 1, limit: 20 };
const queryString = buildQueryString(params); // "?q=javascript&page=1&limit=20"

6. 锚点（Hash/Fragment） ​

  • URL 中 # 后面的部分
  • 格式：#section1、#top
  • 不会发送到服务器
  • 用于页面内跳转（锚点定位）

Hash 的特点：

  • 不会发送到服务器：浏览器不会将 hash 部分发送到服务器
  • 客户端处理：完全由客户端（浏览器）处理
  • 不触发页面刷新：改变 hash 不会重新加载页面
  • 可用于前端路由：单页应用（SPA）常用 hash 路由
  • 可通过 window.location.hash 获取

JavaScript 操作 Hash：

javascript
// 获取 hash
const hash = window.location.hash; // "#section1"

// 设置 hash
window.location.hash = 'section2';

// 监听 hash 变化
window.addEventListener('hashchange', function(event) {
  console.log('Hash 变化:', window.location.hash);
  console.log('旧 URL:', event.oldURL);
  console.log('新 URL:', event.newURL);
});

// 移除 hash
window.location.hash = '';

Hash 路由示例（前端路由）：

javascript
// Hash 路由实现
class HashRouter {
  constructor() {
    this.routes = {};
    this.currentHash = '';
    
    window.addEventListener('load', () => this.handleRoute());
    window.addEventListener('hashchange', () => this.handleRoute());
  }
  
  route(path, callback) {
    this.routes[path] = callback;
  }
  
  handleRoute() {
    this.currentHash = window.location.hash.slice(1) || '/';
    const callback = this.routes[this.currentHash];
    if (callback) {
      callback();
    }
  }
}

// 使用
const router = new HashRouter();
router.route('/', () => console.log('首页'));
router.route('/about', () => console.log('关于页'));
router.route('/contact', () => console.log('联系页'));

URL 编码 ​

URL 中某些字符有特殊含义，需要进行编码：

保留字符（需要编码）：

  • : / ? # [ ] @ ! $ & ' ( ) * + , ; =

编码方法：

javascript
// encodeURI：编码整个 URL（保留 URL 结构字符）
encodeURI('https://example.com/path?name=John Doe');
// "https://example.com/path?name=John%20Doe"

// encodeURIComponent：编码 URL 组件（编码所有特殊字符）
encodeURIComponent('John Doe');
// "John%20Doe"

// decodeURI：解码整个 URL
decodeURI('https://example.com/path?name=John%20Doe');
// "https://example.com/path?name=John Doe"

// decodeURIComponent：解码 URL 组件
decodeURIComponent('John%20Doe');

URL 参数总结 ​

Query 参数（查询参数）：

  • 位置：? 后面
  • 格式：?key=value&key2=value2
  • 用途：过滤、排序、分页等查询条件
  • 发送到服务器：是
  • 示例：/api/users?page=1&limit=10&sort=name

Path 参数（路径参数）：

  • 位置：URL 路径中
  • 格式：/api/users/:id
  • 用途：标识资源
  • 发送到服务器：是
  • 示例：/api/users/123

Hash（锚点）：

  • 位置：# 后面
  • 格式：#section1
  • 用途：页面内跳转、前端路由
  • 发送到服务器：否（仅客户端处理）
  • 示例：/page#section1、/#/users/123（Hash 路由）

请求头（Request Headers） ​

域名/资源 ​

  • Host：指定服务器的域名和端口号
  • Referer：表示请求来源页面
  • Origin：请求的源地址（协议+域名+端口）

方法（HTTP Methods） ​

  • GET：获取资源，幂等，可缓存
  • POST：提交数据，创建资源，非幂等
  • PUT：更新资源，幂等
  • DELETE：删除资源，幂等
  • PATCH：部分更新资源
  • HEAD：获取响应头，不返回响应体
  • OPTIONS：获取服务器支持的请求方法（CORS 预检）

简单请求（Simple Request） ​

满足以下所有条件的请求：

  • 方法为：GET、HEAD、POST
  • 请求头只包含：Accept、Accept-Language、Content-Language、Content-Type
  • Content-Type 为：text/plain、multipart/form-data、application/x-www-form-urlencoded

Content-Type ​

  • text/plain：纯文本
  • text/html：HTML 文档
  • application/json：JSON 数据
  • application/xml：XML 数据
  • application/x-www-form-urlencoded：表单数据
  • multipart/form-data：文件上传
  • application/octet-stream：二进制流

其他重要请求头 ​

  • Authorization：身份认证信息（Bearer Token、Basic Auth）
  • Cookie：客户端发送的 Cookie
  • User-Agent：客户端信息
  • Accept：客户端可接受的响应类型
  • Accept-Encoding：支持的压缩算法（gzip、deflate、br）
  • Cache-Control：缓存控制
  • If-None-Match：条件请求（ETag）
  • If-Modified-Since：条件请求（时间戳）

响应头（Response Headers） ​

源（Origin） ​

  • Access-Control-Allow-Origin：允许跨域的源
  • Access-Control-Allow-Methods：允许的 HTTP 方法
  • Access-Control-Allow-Headers：允许的请求头
  • Access-Control-Allow-Credentials：是否允许携带凭证
  • Access-Control-Max-Age：预检请求缓存时间

状态码（Status Codes） ​

  • 1xx：信息性状态码
    • 100 Continue：继续
    • 101 Switching Protocols：协议切换
  • 2xx：成功状态码
    • 200 OK：请求成功
    • 201 Created：资源已创建
    • 204 No Content：无内容
    • 206 Partial Content：部分内容
  • 3xx：重定向状态码
    • 301 Moved Permanently：永久重定向
    • 302 Found：临时重定向
    • 304 Not Modified：未修改（使用缓存）
  • 4xx：客户端错误
    • 400 Bad Request：请求错误
    • 401 Unauthorized：未授权
    • 403 Forbidden：禁止访问
    • 404 Not Found：资源不存在
    • 405 Method Not Allowed：方法不允许
    • 408 Request Timeout：请求超时
    • 413 Payload Too Large：请求体过大
    • 414 URI Too Long：URI 过长
  • 5xx：服务器错误
    • 500 Internal Server Error：服务器内部错误
    • 502 Bad Gateway：网关错误
    • 503 Service Unavailable：服务不可用
    • 504 Gateway Timeout：网关超时

缓存相关响应头 ​

  • Cache-Control：缓存控制指令
    • max-age=3600：缓存有效期
    • no-cache：需要重新验证
    • no-store：不缓存
    • private：仅客户端缓存
    • public：可被代理服务器缓存
  • ETag：资源版本标识符
  • Last-Modified：资源最后修改时间
  • Expires：资源过期时间（HTTP/1.0）

其他重要响应头 ​

  • Content-Type：响应体类型
  • Content-Length：响应体长度
  • Content-Encoding：内容编码（gzip、deflate、br）
  • Set-Cookie：设置 Cookie
  • Location：重定向地址
  • Server：服务器信息
  • X-Frame-Options：防止点击劫持
  • X-Content-Type-Options：防止 MIME 类型嗅探
  • Strict-Transport-Security：强制 HTTPS

跨域（CORS） ​

是什么原因造成的 ​

  • 同源策略（Same-Origin Policy）：浏览器安全机制
  • 同源定义：协议、域名、端口完全相同
  • 跨域场景：
    • 不同域名：http://a.com 访问 http://b.com
    • 不同端口：http://localhost:3000 访问 http://localhost:8080
    • 不同协议：http://example.com 访问 https://example.com
    • 子域名：http://www.example.com 访问 http://api.example.com

  • 原理：利用 <script> 标签不受同源策略限制

三种方法如何选择，缺陷是什么，适用于什么场景 ​

  1. JSONP

    • 适用：简单 GET 请求，老浏览器兼容
    • 缺陷：只支持 GET，安全性差
    • 场景：第三方 API 调用（如天气、地图）

  2. CORS

    • 适用：现代浏览器，需要完整 HTTP 方法支持
    • 缺陷：需要服务器配置，预检请求增加延迟
    • 场景：前后端分离，API 服务

  3. 代理

    • 适用：开发环境，无法修改服务器配置
    • 缺陷：增加服务器负载，需要额外配置
    • 场景：开发调试，统一 API 网关

function request(options) {
  return new Promise((resolve, reject) => {
    const xhr = new XMLHttpRequest();
    xhr.open(options.method || 'GET', options.url, true);
    
    // 设置请求头
    if (options.headers) {
      Object.keys(options.headers).forEach(key => {
        xhr.setRequestHeader(key, options.headers[key]);
      });
    }
    
    // 设置超时
    if (options.timeout) {
      xhr.timeout = options.timeout;
    }
    
    xhr.onload = function() {
      if (xhr.status >= 200 && xhr.status < 300) {
        try {
          const data = JSON.parse(xhr.responseText);
          resolve(data);
        } catch (e) {
          resolve(xhr.responseText);
        }
      } else {
        reject(new Error(`HTTP ${xhr.status}: ${xhr.statusText}`));
      }
    };
    
    xhr.onerror = function() {
      reject(new Error('Network error'));
    };
    
    xhr.ontimeout = function() {
      reject(new Error('Request timeout'));
    };
    
    xhr.send(options.data ? JSON.stringify(options.data) : null);
  });
}

class HttpClient {
  constructor(baseURL = '', defaultOptions = {}) {
    this.baseURL = baseURL;
    this.defaultOptions = {
      headers: {
        'Content-Type': 'application/json',
        ...defaultOptions.headers
      },
      ...defaultOptions
    };
  }
  
  async request(url, options = {}) {
    const config = {
      ...this.defaultOptions,
      ...options,
      headers: {
        ...this.defaultOptions.headers,
        ...options.headers
      }
    };
    
    const fullURL = `${this.baseURL}${url}`;
    
    try {
      const response = await fetch(fullURL, config);
      
      if (!response.ok) {
        throw new Error(`HTTP ${response.status}: ${response.statusText}`);
      }
      
      const contentType = response.headers.get('content-type');
      if (contentType && contentType.includes('application/json')) {
        return await response.json();
      }
      return await response.text();
    } catch (error) {
      console.error('Request failed:', error);
      throw error;
    }
  }
  
  get(url, options = {}) {
    return this.request(url, { ...options, method: 'GET' });
  }
  
  post(url, data, options = {}) {
    return this.request(url, {
      ...options,
      method: 'POST',
      body: JSON.stringify(data)
    });
  }
  
  put(url, data, options = {}) {
    return this.request(url, {
      ...options,
      method: 'PUT',
      body: JSON.stringify(data)
    });
  }
  
  delete(url, options = {}) {
    return this.request(url, { ...options, method: 'DELETE' });
  }
}

TCP/IP 协议 ​

TCP（Transmission Control Protocol） ​

TCP 特点 ​

  • 面向连接：需要三次握手建立连接
  • 可靠传输：保证数据顺序和完整性
  • 流量控制：滑动窗口机制，防止发送方发送数据过快
  • 拥塞控制：慢启动、拥塞避免、快重传、快恢复
  • 全双工通信：双方可以同时发送和接收数据

TCP 三次握手（建立连接） ​

TCP 三次握手是建立可靠连接的过程，确保双方都能发送和接收数据：

第一次握手（SYN）：

  • 客户端向服务器发送 SYN 包（SYN=1, seq=x）
  • 客户端进入 SYN_SENT 状态
  • 目的：客户端告诉服务器"我想建立连接，我的初始序列号是 x"

第二次握手（SYN-ACK）：

  • 服务器收到 SYN 包后，向客户端发送 SYN-ACK 包（SYN=1, ACK=1, seq=y, ack=x+1）
  • 服务器进入 SYN_RCVD 状态
  • 目的：服务器告诉客户端"我收到了你的请求，同意建立连接，我的初始序列号是 y，期待你的下一个序号是 x+1"

第三次握手（ACK）：

  • 客户端收到 SYN-ACK 包后，向服务器发送 ACK 包（ACK=1, seq=x+1, ack=y+1）
  • 客户端和服务器都进入 ESTABLISHED 状态
  • 目的：客户端告诉服务器"我收到了你的确认，连接已建立"

状态变化流程：

客户端：CLOSED → SYN_SENT → ESTABLISHED
服务器：CLOSED → LISTEN → SYN_RCVD → ESTABLISHED
1
2

为什么需要三次握手？

  1. 防止历史连接：如果客户端发送的 SYN 包因为网络延迟，在连接关闭后才到达服务器，服务器会误认为这是新的连接请求。通过三次握手，客户端可以在收到服务器的响应后判断这是否是历史连接，如果是则发送 RST 包终止连接。
  2. 同步双方序列号：TCP 需要序列号来保证数据的可靠传输，三次握手确保双方都知道对方的初始序列号。
  3. 避免资源浪费：如果只有两次握手，服务器在收到 SYN 后就建立连接并分配资源，但客户端可能因为网络问题没有收到服务器的响应，导致服务器资源浪费。

TCP 四次挥手（断开连接） ​

TCP 四次挥手是断开连接的过程，确保双方都能正常关闭连接：

第一次挥手（FIN）：

  • 客户端向服务器发送 FIN 包（FIN=1, seq=u）
  • 客户端进入 FIN_WAIT_1 状态
  • 目的：客户端告诉服务器"我没有数据要发送了，准备关闭连接"

第二次挥手（ACK）：

  • 服务器收到 FIN 包后，向客户端发送 ACK 包（ACK=1, ack=u+1）
  • 服务器进入 CLOSE_WAIT 状态，客户端进入 FIN_WAIT_2 状态
  • 目的：服务器告诉客户端"我收到了你的关闭请求，但我可能还有数据要发送"

第三次挥手（FIN）：

  • 服务器发送完所有数据后，向客户端发送 FIN 包（FIN=1, seq=w）
  • 服务器进入 LAST_ACK 状态
  • 目的：服务器告诉客户端"我也没有数据要发送了，准备关闭连接"

第四次挥手（ACK）：

  • 客户端收到 FIN 包后，向服务器发送 ACK 包（ACK=1, ack=w+1）
  • 客户端进入 TIME_WAIT 状态，等待 2MSL（Maximum Segment Lifetime）后进入 CLOSED 状态
  • 服务器收到 ACK 后立即进入 CLOSED 状态
  • 目的：客户端告诉服务器"我收到了你的关闭请求，连接已关闭"

状态变化流程：

客户端：ESTABLISHED → FIN_WAIT_1 → FIN_WAIT_2 → TIME_WAIT → CLOSED
服务器：ESTABLISHED → CLOSE_WAIT → LAST_ACK → CLOSED
1
2

为什么需要四次挥手？

  • TCP 是全双工通信，双方可以独立关闭自己的发送通道
  • 当一方发送 FIN 时，只是表示这一方不再发送数据，但还可以接收数据
  • 另一方可能还有数据要发送，所以先发送 ACK 确认收到 FIN，等数据发送完毕后再发送 FIN

为什么客户端需要等待 2MSL？

  1. 确保最后一个 ACK 到达服务器：如果客户端的 ACK 丢失，服务器会重传 FIN，客户端需要能够接收并响应
  2. 防止旧连接的数据包影响新连接：等待 2MSL 确保网络中所有旧连接的数据包都消失，避免影响新的连接（MSL 通常是 30 秒到 2 分钟，所以 TIME_WAIT 状态通常持续 1 到 4 分钟）

WebSocket 重连机制 ​

javascript
class WebSocketClient {
  constructor(url, options = {}) {
    this.url = url;
    this.options = {
      reconnectInterval: 1000,
      maxReconnectInterval: 30000,
      reconnectDecay: 1.5,
      timeoutInterval: 2000,
      maxReconnectAttempts: null,
      ...options
    };
    this.reconnectAttempts = 0;
    this.shouldReconnect = true;
    this.connect();
  }
  
  connect() {
    this.ws = new WebSocket(this.url);
    
    this.ws.onopen = (event) => {
      console.log('WebSocket 连接已建立');
      this.reconnectAttempts = 0;
      this.onopen && this.onopen(event);
    };
    
    this.ws.onmessage = (event) => {
      this.onmessage && this.onmessage(event);
    };
    
    this.ws.onerror = (error) => {
      console.error('WebSocket 错误:', error);
      this.onerror && this.onerror(error);
    };
    
    this.ws.onclose = (event) => {
      console.log('WebSocket 连接已关闭');
      this.onclose && this.onclose(event);
      
      if (this.shouldReconnect) {
        this.reconnect();
      }
    };
  }
  
  reconnect() {
    if (this.options.maxReconnectAttempts && 
        this.reconnectAttempts >= this.options.maxReconnectAttempts) {
      console.error('达到最大重连次数');
      return;
    }
    
    const timeout = Math.min(
      this.options.reconnectInterval * Math.pow(this.options.reconnectDecay, this.reconnectAttempts),
      this.options.maxReconnectInterval
    );
    
    setTimeout(() => {
      this.reconnectAttempts++;
      console.log(`尝试重连 (${this.reconnectAttempts})...`);
      this.connect();
    }, timeout);
  }
  
  send(data) {
    if (this.ws.readyState === WebSocket.OPEN) {
      this.ws.send(typeof data === 'string' ? data : JSON.stringify(data));
    } else {
      console.error('WebSocket 未连接');
    }
  }
  
  close() {
    this.shouldReconnect = false;
    this.ws.close();
  }
}

SSE（Server-Sent Events） ​

SSE 是什么 ​

  • 服务器推送事件（Server-Sent Events）

SSE 应用场景 ​

  • 实时通知
  • 股票行情

class SSEClient {
  constructor(url, options = {}) {
    this.url = url;
    this.options = {
      withCredentials: false,
      ...options
    };
    this.eventSource = null;
    this.listeners = new Map();
    this.connect();
  }
  
  connect() {
    this.eventSource = new EventSource(this.url, {
      withCredentials: this.options.withCredentials
    });
    
    this.eventSource.onopen = () => {
      console.log('SSE 连接已建立');
      this.onopen && this.onopen();
    };
    
    this.eventSource.onmessage = (event) => {
      try {
        const data = JSON.parse(event.data);
        this.onmessage && this.onmessage(data, event);
      } catch (e) {
        this.onmessage && this.onmessage(event.data, event);
      }
    };
    
    this.eventSource.onerror = (error) => {
      console.error('SSE 错误:', error);
      this.onerror && this.onerror(error);
    };
    
    // 注册自定义事件监听器
    this.listeners.forEach((handler, eventType) => {
      this.eventSource.addEventListener(eventType, handler);
    });
  }
  
  addEventListener(eventType, handler) {
    this.listeners.set(eventType, handler);
    if (this.eventSource) {
      this.eventSource.addEventListener(eventType, handler);
    }
  }
  
  removeEventListener(eventType) {
    this.listeners.delete(eventType);
    if (this.eventSource) {
      this.eventSource.removeEventListener(eventType, 
        this.listeners.get(eventType));
    }
  }
  
  close() {
    if (this.eventSource) {
      this.eventSource.close();
      this.eventSource = null;
    }
  }
}

缓存机制 ​

浏览器缓存概述 ​

浏览器缓存是提高网站性能的重要手段，可以减少网络请求，加快页面加载速度。缓存分为强缓存和协商缓存两种。

强缓存（Strong Cache） ​

强缓存是指浏览器直接从本地缓存中读取资源，不向服务器发送请求。

Cache-Control（HTTP/1.1） ​

Cache-Control 是最重要的缓存控制头，优先级高于 Expires。

常用指令：

  • max-age=<seconds>：缓存有效期（秒），如 max-age=3600 表示缓存 1 小时
  • no-cache：需要重新验证，每次使用前必须向服务器验证
  • no-store：不缓存，完全不存储任何缓存
  • private：仅客户端缓存，代理服务器不能缓存
  • public：可被任何缓存（客户端、代理服务器）缓存
  • must-revalidate：缓存过期后必须向服务器验证
  • immutable：资源不会改变，浏览器可以永久缓存
  • s-maxage=<seconds>：代理服务器的缓存时间

示例：

http
Cache-Control: public, max-age=31536000, immutable
1

Expires（HTTP/1.0） ​

Expires 指定资源的过期时间，是绝对时间。

示例：

http
Expires: Wed, 21 Oct 2025 07:28:00 GMT
1

注意：Expires 已被 Cache-Control 的 max-age 取代，因为 Expires 依赖客户端和服务器的时间同步。

协商缓存（Negotiation Cache） ​

协商缓存是指浏览器向服务器发送请求，服务器验证资源是否更新，如果未更新则返回 304，浏览器使用缓存。

ETag + If-None-Match ​

ETag（Entity Tag）是服务器为资源生成的唯一标识符，通常是内容的哈希值。

流程：

  1. 服务器返回资源时，在响应头中添加 ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"
  2. 浏览器再次请求时，在请求头中添加 If-None-Match: "33a64df551425fcc55e4d42a148795d9f25f89d4"
  3. 服务器比较 ETag：
    • 如果匹配，返回 304 Not Modified，不返回响应体
    • 如果不匹配，返回 200 OK 和新资源

优点：

  • 精确：可以检测到内容的任何变化
  • 支持弱验证：W/"weak-etag" 表示语义等价即可

示例：

http
请求：
GET /api/resource HTTP/1.1
If-None-Match: "33a64df551425fcc55e4d42a148795d9f25f89d4"

响应（未修改）：
HTTP/1.1 304 Not Modified
ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"

Last-Modified + If-Modified-Since ​

Last-Modified 是资源的最后修改时间。

流程：

  1. 服务器返回资源时，在响应头中添加 Last-Modified: Wed, 21 Oct 2015 07:28:00 GMT
  2. 浏览器再次请求时，在请求头中添加 If-Modified-Since: Wed, 21 Oct 2015 07:28:00 GMT
  3. 服务器比较时间：
    • 如果未修改，返回 304 Not Modified
    • 如果已修改，返回 200 OK 和新资源

缺点：

  • 精度问题：只能精确到秒，1 秒内的多次修改无法检测
  • 时间同步问题：依赖服务器和客户端时间同步

  2. HTML：使用 no-cache，总是验证
  3. API：根据数据特性选择缓存策略

Cookie ​

Cookie 概述 ​

Cookie 是服务器发送到浏览器并保存在本地的小型数据片段，浏览器会在每次请求时自动携带 Cookie。Cookie 既是存储方式，也是网络协议的一部分，因此在这里详细介绍。

Cookie 特点 ​

  • 存储在客户端：浏览器本地存储
  • 自动携带：每次 HTTP 请求自动发送到服务器
  • 大小限制：单个 Cookie 最大 4KB，每个域名最多 20 个 Cookie（浏览器差异）
  • 域名限制：只能被设置 Cookie 的域名及其子域名访问
  • 路径限制：可以设置 Path 限制 Cookie 的作用路径

Cookie 属性详解 ​

1. Domain（域名）

http
Set-Cookie: name=value; Domain=.example.com
1
  • 指定哪些域名可以访问该 Cookie
  • .example.com 表示 example.com 及其所有子域名都可以访问
  • 如果不设置，默认为当前域名

2. Path（路径）

http
Set-Cookie: name=value; Path=/api
1
  • 指定 Cookie 的作用路径
  • /api 表示只有 /api 及其子路径可以访问
  • 默认为 /，表示整个域名都可以访问

3. Expires（过期时间）

http
Set-Cookie: name=value; Expires=Wed, 21 Oct 2025 07:28:00 GMT
1
  • 指定 Cookie 的过期时间（绝对时间）
  • 过期后浏览器会自动删除

4. Max-Age（最大存活时间）

http
Set-Cookie: name=value; Max-Age=3600
1
  • 指定 Cookie 的存活时间（秒）
  • 3600 表示 1 小时后过期
  • 优先级高于 Expires

5. HttpOnly

http
Set-Cookie: name=value; HttpOnly
1
  • 禁止 JavaScript 访问该 Cookie
  • 只能通过 HTTP 请求发送
  • 防止 XSS 攻击：即使网站被 XSS 攻击，也无法通过 JavaScript 窃取 Cookie

6. Secure

http
Set-Cookie: name=value; Secure
1
  • 只能通过 HTTPS 传输
  • HTTP 请求不会发送该 Cookie
  • 提高安全性：防止 Cookie 在传输过程中被窃取

7. SameSite

http
Set-Cookie: name=value; SameSite=Strict
1
  • 控制跨站请求时是否发送 Cookie
  • Strict：完全禁止跨站携带，最安全
  • Lax：GET 请求可以跨站携带，POST 请求禁止（默认值）
  • None：允许跨站携带，但必须设置 Secure

其他重要概念 ​

负载均衡 ​

  • 轮询（Round Robin）：依次分配请求到各个服务器
  • 加权轮询（Weighted Round Robin）：根据服务器性能分配，性能好的服务器分配更多请求
  • 最少连接（Least Connections）：分配给当前连接数最少的服务器
  • IP 哈希（IP Hash）：根据客户端 IP 地址的哈希值分配，保证同一客户端访问同一服务器
  • 响应时间（Response Time）：根据服务器响应时间分配，响应快的服务器优先

CDN（Content Delivery Network） ​

  • 分布式服务器网络：在全球多个节点部署服务器
  • 就近访问：用户访问最近的 CDN 节点，减少延迟
  • 减少服务器负载：将静态资源分发到 CDN，减轻源服务器压力
  • 提高可用性：即使某个节点故障，其他节点仍可提供服务
  • 缓存加速：CDN 节点缓存内容，提高访问速度

反向代理 ​

  • 隐藏真实服务器：客户端不知道后端服务器的真实地址
  • 负载均衡：将请求分发到多个后端服务器
  • SSL 终止：在反向代理处处理 SSL/TLS，减轻后端服务器负担
  • 缓存静态资源：缓存常用资源，减少后端请求
  • 安全防护：可以作为防火墙，过滤恶意请求
  • 压缩响应：压缩响应内容，减少带宽使用

正向代理 ​

  • 隐藏客户端身份：服务器不知道客户端的真实 IP
  • 访问控制：可以限制访问某些网站
  • 缓存加速：缓存常用内容，提高访问速度
  • 突破访问限制：可以访问被限制的网站
  • 日志记录：记录客户端访问记录

RESTful API ​

  • REST：Representational State Transfer，表现层状态转换
  • 资源：URL 表示资源，HTTP 方法表示操作
  • 无状态：每个请求都是独立的，不依赖之前的请求
  • 统一接口：使用标准的 HTTP 方法（GET、POST、PUT、DELETE）
  • 分层系统：客户端、服务器、代理等可以分层
  • 缓存：支持缓存机制

RESTful API 设计原则：

GET    /api/users           → 获取用户列表
GET    /api/users/123       → 获取 ID 为 123 的用户
POST   /api/users           → 创建用户
PUT    /api/users/123       → 更新 ID 为 123 的用户（完整更新）
PATCH  /api/users/123       → 更新 ID 为 123 的用户（部分更新）
DELETE /api/users/123      → 删除 ID 为 123 的用户

  • 查询语言：客户端可以精确指定需要的数据
  • 单一端点：所有请求都发送到同一个端点
  • 类型系统：强类型系统，支持类型检查
  • 实时订阅：支持实时数据订阅

gRPC ​

  • 高性能 RPC 框架：基于 HTTP/2，性能更好
  • Protocol Buffers：使用 protobuf 进行序列化，体积小、速度快
  • 流式传输：支持流式请求和响应
  • 多语言支持：支持多种编程语言
  • 服务发现：内置服务发现机制

HTTP 管道化（HTTP Pipelining） ​

  • 并发请求：在同一个连接上发送多个请求，无需等待响应
  • 提高效率：减少网络延迟
  • 浏览器支持：现代浏览器默认禁用，因为可能导致队头阻塞问题
  • HTTP/2 替代：HTTP/2 的多路复用解决了管道化的问题

HTTP 压缩 ​

  • Gzip：最常用的压缩算法，压缩率高
  • Brotli：Google 开发的压缩算法，压缩率比 Gzip 更高
  • Deflate：较老的压缩算法
  • 压缩响应：服务器压缩响应内容，客户端解压
  • Accept-Encoding：客户端通过请求头告知支持的压缩算法
  • Content-Encoding：服务器通过响应头告知使用的压缩算法

HTTP 范围请求（Range Request） ​

  • 断点续传：支持从指定位置开始下载
  • Range 头：Range: bytes=0-499 请求前 500 字节
  • 206 状态码：部分内容响应
  • Content-Range：响应头告知内容范围
  • 多线程下载：可以同时请求不同范围，实现多线程下载

HTTP 条件请求 ​

  • If-Match：只有当资源的 ETag 匹配时才处理请求
  • If-None-Match：只有当资源的 ETag 不匹配时才处理请求
  • If-Modified-Since：只有当资源在指定时间后修改过才处理请求
  • If-Unmodified-Since：只有当资源在指定时间后未修改过才处理请求
  • 304 Not Modified：资源未修改，使用缓存
