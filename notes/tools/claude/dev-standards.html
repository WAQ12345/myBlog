Dev Standards (Merged) ​

Part 1: Dev Rules ​

  • 交互：回复中文，称呼「亲爱的开发者：」；复杂需求拆小任务；实现前后检查类型与 props；扩展功能保持兼容、不额外加功能/依赖。
  • 质量：遵循项目架构与 TypeScript/ESLint；单一职责；第一性原理、KISS、SOLID；优先复用组件与 hooks；复杂逻辑加注释，props 类型完整；变更范围最小化；API 有疑问先问。
  • 文档：不自动生成修复/验证/总结文档；仅当用户明确要求「写文档」等时才生成；默认分析问题、修复、简要说明。
  • 自动化：可自动执行类型检查、ESLint、常规命令；重要变更（构建、路由）需确认；重要操作前保留副本；API 变更优先改 TS 接口；大改前分析依赖与影响。
  • 优化：优化未使用 import、重复样式；性能建议；异常与加载状态；内存与异步竞态；复用与架构梳理；设计模式与状态管理。
  • 测试与追溯：关键组件与函数单测；提交信息清晰；重大 UI 变更可配变更文档；API/props 变更提供兼容或迁移；修改后任务总结。
  • 安全与体验：输入验证与 XSS/CSRF；不暴露敏感信息与内部错误；懒加载与代码分割；交互反馈；错误边界；无障碍与响应式。
  • 协作：重要变更前代码审查；沟通与知识分享；持续改进。

Part 2: Comment Standards ​

  • 原则：必要才写；不用 emoji；与代码同步。
  • 何时写：复杂逻辑、业务含义、公共 API（JSDoc）、临时方案/TODO、性能与副作用说明。
  • 何时不写：自解释命名、重复函数名的单行说明、简单 getter/setter。
  • 格式：单行 //；多行块注释或 JSDoc；公共函数与组件写 @param、@returns、@example。
  • 语言：与项目约定一致；同一项目内统一。

Part 3: Frontend Standards ​

  • 通用：遵循项目架构与 ESLint；组件单一职责，props 类型完整；复用优先；变更范围最小化；API/props 变更需兼容或迁移说明。
  • 组件与状态：合理拆分；状态就近；异步与副作用注意清理与竞态；表单与输入校验与防重复。
  • 样式与布局：与设计系统一致；响应式与无障碍。
  • 性能：懒加载与代码分割；长列表虚拟列表或分页；合理使用 memo/useMemo/useCallback。
  • 安全与错误：输入校验与转义；不暴露敏感信息；错误边界与加载/错误状态。
  • 测试：关键组件与逻辑单测；E2E 按需；公共组件与 API 有说明或 JSDoc。
