Cursor 开发规则 ​

基础交互规则 ​

  1. 语言规范：默认情况下，所有回复都必须是中文，而且需要在开头称呼用户为"亲爱的开发者："
  2. 任务分解：复杂需求拆解成小任务，分步实现，每完成一个小任务后再继续
  3. 代码检查：代码实现前后要仔细检查，确保类型定义完整、组件 props 正确
  4. 功能扩展原则：在已有功能基础上添加新功能时，必须确保：
    • 不影响原有功能和组件复用性
    • 不添加其他功能、代码、逻辑、文件、配置、依赖
    • 保持向后兼容性

代码质量与架构 ​

  1. 代码规范：遵循项目架构设计，保持代码风格与 TypeScript/ESLint 规范一致
  2. 单一职责：组件设计遵循单一职责原则，不混合多个变更
  3. 设计原则：在进行组件设计规划时，符合"第一性原理"
  4. 实现原则：在代码实现时，符合"KISS原则"和"SOLID原则"
  5. 代码复用：优先使用现有组件库和 hooks，避免重复代码
  6. 依赖管理：不引入不必要的依赖，优先使用项目已有库
  7. 代码可读性：确保代码可读性，复杂逻辑添加注释，组件 props 类型详细定义
  8. 变更范围：代码变更范围最小化，避免修改公共组件、全局状态
  9. 逻辑验证：实现后进行基本逻辑自检，确保状态管理与生命周期正确
  10. API 设计：如有疑问，先询问再修改，不要擅自改变组件 API 设计

文档与输出规则 ​

  1. 文档生成策略：用户不希望在解决问题后自动生成修复文档、验证指南、总结文档等文档
  2. 文档触发条件：只有当用户明确要求"写文档"、"生成文档"、"需要文档"、"写个验证指南"等关键词时，才生成相应文档
  3. 默认行为：默认行为应该是：分析问题、修复代码、简要说明修复内容，但不生成任何文档文件

自动化执行与安全策略 ​

  1. 自动执行操作：自动执行无需严格确认的操作，提高效率：
  • 自动执行 TypeScript 类型检查、ESLint 验证
  • 文件操作（创建组件、修改样式文件）无需额外确认
  • 常规命令（如 npm install、启动开发服务器）可直接执行
  • 涉及构建配置、路由修改等重要变更仍需确认
  1. 重要操作保护：重要操作（修改全局状态、路由配置）应先保留副本
  2. API 接口变更：涉及 API 接口变更，优先修改 TypeScript 接口定义
  3. 影响范围分析：执行影响较大的修改前，自动检测组件依赖关系，分析影响范围

代码质量优化 ​

  1. 代码优化：代码生成后，自动优化（移除未使用 imports、合并重复样式）
  2. 性能优化：对可能影响性能的代码（如不必要的重渲染、大型循环）提供优化建议
  3. 错误处理：确保异常处理和加载状态管理，防止白屏和渲染错误
  4. 内存管理：注意内存泄漏问题，及时清理事件监听器、定时器等资源
  5. 异步处理：正确处理异步操作，避免竞态条件和状态不一致

架构感知与设计 ​

  1. 组件复用：优先分析现有组件库与状态管理模式，避免创建冗余组件
  2. 功能复用：添加功能时，优先考虑复用 hooks、contexts 或现有组件
  3. 架构梳理：如遇架构不清晰，先梳理组件层次与数据流，再执行修改
  4. 设计模式：合理使用设计模式，提高代码的可维护性和扩展性
  5. 状态管理：合理设计状态管理，避免状态冗余和循环依赖

测试与验证 ​

  1. 单元测试：为关键组件和函数编写单元测试
  2. 集成测试：确保组件间的集成正常工作
  3. 边界测试：测试边界条件和异常情况
  4. 性能测试：关注组件渲染性能和内存使用

代码变更的可追溯性 ​

  1. 提交信息：提供清晰的 commit 信息，描述组件变更和影响范围
  2. 变更文档：对于 UI 组件重大调整，生成变更文档与截图对比
  3. 兼容性方案：API 或 props 变更时，提供向下兼容方案或迁移指南
  4. 项目分析：执行任务前，先分析项目结构和组件关系文档
  5. 任务总结：每次修改后，生成任务总结，说明组件变更和状态管理调整
  6. 文档维护：手动维护组件文档与设计系统说明，确保长期可维护性

安全与最佳实践 ​

  1. 输入验证：对所有用户输入进行验证和清理
  2. XSS 防护：避免直接插入用户内容到 DOM 中
  3. CSRF 防护：在涉及状态变更的操作中使用 CSRF 令牌
  4. 敏感信息：不在客户端代码中暴露敏感信息（API 密钥、密码等）
  5. 错误信息：不向用户暴露系统内部错误信息

性能与用户体验 ​

  1. 加载优化：实现懒加载和代码分割，减少初始加载时间
  2. 交互反馈：为用户操作提供及时的视觉反馈
  3. 错误边界：实现错误边界，防止单个组件错误影响整个应用
  4. 无障碍访问：确保组件支持键盘导航和屏幕阅读器
  5. 响应式设计：确保组件在不同屏幕尺寸下正常工作

团队协作 ​

  1. 代码审查：重要变更前进行代码审查
  2. 沟通协作：与团队成员保持良好沟通，及时同步变更信息
  3. 知识分享：将解决方案和最佳实践分享给团队
  4. 持续改进：根据项目发展持续优化开发流程和规范
